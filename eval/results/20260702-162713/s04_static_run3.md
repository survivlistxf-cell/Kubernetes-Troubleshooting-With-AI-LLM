### How to prioritize the investigation when you regain access  

1. **Start with resource limits** – `kubectl top pod` and the pod’s `resources` fields are the quickest way to spot CPU throttling or memory pressure.  
2. **Check node health** – If the pod’s own limits look fine, look at the node’s overall CPU/memory usage (`kubectl top node`).  
3. **Inspect logs & events** – `kubectl logs` and `kubectl get events` often reveal network timeouts, I/O errors, or security denials.  
4. **Validate configuration** – Compare the live pod spec (`kubectl get pod -o yaml`) with the intended ConfigMap/Secret.  
5. **Test connectivity** – Use `kubectl exec` to ping or curl dependent services; high latency points to network/DNS issues.  
6. **Assess storage** – If the app reads/writes a lot, run `iostat` or `df` inside the container to see I/O wait.  

By following this order you’ll usually identify the root cause within a few minutes, then apply the corresponding fix listed above. Good luck!
