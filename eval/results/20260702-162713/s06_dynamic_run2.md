## Most likely reasons why the **s06‑limited‑sa** service account cannot list Secrets  

| # | Probable cause | Why it happens | How to verify (run when you have cluster access) | How to fix |
|---|----------------|----------------|--------------------------------------------------|------------|
| 1 | **Missing RBAC permission** – the ServiceAccount is not bound to a Role/ClusterRole that allows `list` (or `get`) on `secrets` in the `kubexplain-eval` namespace. | By default a ServiceAccount only has the permissions granted by the `system:serviceaccount:*:*` group, which does **not** include secret enumeration. | ```bash\n# Does the SA have the right to list secrets?\nkubectl auth can-i list secrets --as=system:serviceaccount:kubexplain-eval:s06-limited-sa -n kubexplain-eval\n```<br>If the output is `no`, the permission is missing. | Create (or update) a Role that grants `list`/`get` on `secrets` and bind it to the SA:<br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: Role\nmetadata:\n  name: secret‑reader\n  namespace: kubexplain-eval\nrules:\n- apiGroups: [\"\"]\n  resources: [\"secrets\"]\n  verbs: [\"get\",\"list\"]\n---\napiVersion: rbac.authorization.k8s.io/v1\nkind: RoleBinding\nmetadata:\n  name: secret‑reader‑binding\n  namespace: kubexplain-eval\nsubjects:\n- kind: ServiceAccount\n  name: s06-limited-sa\n  namespace: kubexplain-eval\nroleRef:\n  kind: Role\n  name: secret‑reader\n  apiGroup: rbac.authorization.k8s.io\n```<br>Apply the manifest and re‑test. |
| 2 | **Namespace mismatch** – the code is trying to list Secrets in a different namespace (e.g., `default`) while the Role only covers `kubexplain-eval`. | RBAC rules are namespace‑scoped; a Role in `kubexplain-eval` does not grant access to Secrets elsewhere. | ```bash\n# Try listing secrets in the target namespace the app uses\nkubectl get secrets -n <target‑ns> --as=system:serviceaccount:kubexplain-eval:s06-limited-sa\n```<br>If it works in `kubexplain-eval` but fails in another namespace, the issue is the namespace. | Either: <br>• Adjust the application to request Secrets from its own namespace, **or** <br>• Grant cross‑namespace access by creating a **ClusterRole** with secret permissions and a **ClusterRoleBinding** (or a RoleBinding in the other namespace). Example ClusterRole: <br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: ClusterRole\nmetadata:\n  name: secret‑reader‑cluster\nrules:\n- apiGroups: [\"\"]\n  resources: [\"secrets\"]\n  verbs: [\"get\",\"list\"]\n```<br>Bind it to the SA in the needed namespace(s). |
| 3 | **Secret type restriction** – the SA is allowed to read its own token secret but not other secret types (e.g., `kubernetes.io/dockerconfigjson`). Some clusters enable the `RestrictedSecrets` admission plugin that blocks non‑service‑account secrets. | The admission controller can reject the request before RBAC is even consulted, returning a “Forbidden” error that looks like an RBAC problem. | ```bash\nkubectl api-versions | grep admission\n# Look for a plugin like RestrictedSecrets in the API server config (requires cluster‑admin access)\n```<br>Or check the exact error message; it will mention “restricted secret”. | If the cluster uses `RestrictedSecrets`, you must either: <br>• Request only the allowed secret types (service‑account token), **or** <br>• Ask the cluster admin to add the needed secret type to the allowed list via the `--restricted-secret-types` flag, or disable the plugin if appropriate. |
| 4 | **Projected ServiceAccount token with limited audience** – the pod is using a projected token that only allows access to the API server for a specific audience, and the request to list secrets does not match that audience. | Since Kubernetes 1.22, projected tokens can be scoped; a token that lacks the `kubernetes.io/serviceaccount` audience will be rejected for secret operations. | ```bash\n# Inspect the token volume mount (when you have pod access)\nkubectl exec <pod> -c <container> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token | base64 -d | jq .audiences\n```<br>If the audience list does not contain `api` or the default audience, the token is too narrow. | Update the pod spec to request a token with the needed audience, e.g.:<br>```yaml\nserviceAccountName: s06-limited-sa\nautomountServiceAccountToken: true\n# For projected token:\nvolumes:\n- name: sa-token\n  projected:\n    sources:\n    - serviceAccountToken:\n        audience: api\n```<br>Redeploy the pod. |
| 5 | **NetworkPolicy or PodSecurityPolicy blocking the request** – a NetworkPolicy isolates the pod from the API server endpoint, or a PSP prevents the pod from mounting the service‑account token. | The pod can reach the API server but the request is dropped or the token is not present, leading to a “Forbidden” error when the API server receives an unauthenticated request. | ```bash\n# Check NetworkPolicies that select the pod’s labels\nkubectl get networkpolicy -n kubexplain-eval -o wide\n# Verify the pod has a service‑account token mounted\nkubectl exec <pod> -- ls /var/run/secrets/kubernetes.io/serviceaccount\n``` | Adjust the NetworkPolicy to allow egress to the API server (`kubernetes.default.svc`) and/or modify the PSP (or the newer PodSecurity standards) to permit service‑account token mounting. |
| 6 | **API server mis‑configuration / missing `--service-account-lookup` flag** – the API server is started without the flag that allows service‑account token validation, causing secret‑list calls to be rejected. | Rare, but if the flag is disabled the API server cannot map the token to a ServiceAccount, resulting in a generic permission error. | ```bash\n# Requires cluster‑admin; view the kube‑apiserver pod spec\nkubectl -n kube-system get pod -l component=kube-apiserver -o yaml | grep service-account-lookup\n``` | Re‑enable the flag (`--service-account-lookup=true`) and restart the API server. This usually requires a control‑plane admin. |

### How to prioritize the investigation

1. **Check RBAC first** – it is the most common cause for “cannot list secrets”.  
2. **Confirm the namespace** the application is targeting.  
3. **Look at admission plugins** (RestrictedSecrets) if the error mentions secret type.  
4. **Inspect the token** (projected vs. classic) for audience restrictions.  
5. **Validate NetworkPolicy / PSP** only if the above are all correct.  
6. **API‑server flag** is a last‑resort, cluster‑wide setting.

When you regain access, run the verification commands in the order above; the first one that returns a problem is almost certainly the root cause. Fix the identified issue, redeploy the pod (or restart the deployment), and the secret‑listing operation should succeed.
