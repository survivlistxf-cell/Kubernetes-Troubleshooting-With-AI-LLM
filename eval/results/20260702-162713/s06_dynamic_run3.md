## Most likely reasons the `s06‑limited‑sa` service account cannot list Secrets  

| # | Likely cause | Why it happens | How to verify (when you have cluster access) | How to fix |
|---|--------------|----------------|---------------------------------------------|------------|
| 1 | **Missing RBAC permission** – no `Role`/`ClusterRole` that allows `list`/`get` on `secrets` for this SA. | By default a service‑account only has the permissions granted by the `system:serviceaccount:*:*` group, which does **not** include secret access. | ```bash\nkubectl auth can-i list secrets --as=system:serviceaccount:kubexplain-eval:s06-limited-sa -n kubexplain-eval\n```<br>If the result is `no`, the SA lacks the right. <br>Also inspect existing Role/ClusterRole bindings: <br>```bash\nkubectl get rolebinding,clusterrolebinding -n kubexplain-eval -o yaml | grep s06-limited-sa -C2\n``` | Create (or update) a `Role` that grants the needed verbs and bind it to the SA: <br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: Role\nmetadata:\n  name: secret‑reader\n  namespace: kubexplain-eval\nrules:\n- apiGroups: [\"\"]\n  resources: [\"secrets\"]\n  verbs: [\"get\",\"list\",\"watch\"]\n---\napiVersion: rbac.authorization.k8s.io/v1\nkind: RoleBinding\nmetadata:\n  name: secret‑reader‑binding\n  namespace: kubexplain-eval\nsubjects:\n- kind: ServiceAccount\n  name: s06-limited-sa\n  namespace: kubexplain-eval\nroleRef:\n  kind: Role\n  name: secret‑reader\n  apiGroup: rbac.authorization.k8s.io\n```<br>Apply the manifest (`kubectl apply -f …`). |
| 2 | **Namespace‑scoped permission mismatch** – the SA is trying to list Secrets in a different namespace. | A `Role` only applies to its own namespace. If the code calls the API without specifying the namespace (or uses a different one), the request is denied. | ```bash\nkubectl auth can-i list secrets --as=system:serviceaccount:kubexplain-eval:s06-limited-sa -n <other‑ns>\n```<br>Replace `<other‑ns>` with the namespace you expect the Secrets to live in. | Either: <br>• Grant the SA a **ClusterRole** with secret access and bind it with a `ClusterRoleBinding` (if cross‑namespace access is required). <br>• Or modify the application to request Secrets only from `kubexplain-eval`. |
| 3 | **Service‑account token not mounted / expired** – the pod cannot authenticate to the API server. | If the pod’s token volume is missing, corrupted, or the token has been revoked (e.g., after a SA recreation), the API call fails with a 403/401 before RBAC is even evaluated. | ```bash\nkubectl exec <pod> -n kubexplain-eval -- cat /var/run/secrets/kubernetes.io/serviceaccount/token\n```<br>Check that the file exists and is non‑empty. <br>Also look at the pod’s events: <br>```bash\nkubectl describe pod <pod> -n kubexplain-eval | grep -i token\n``` | Re‑create the pod (or delete & let the Deployment/ReplicaSet recreate it) so the kubelet injects a fresh token. If the SA was recreated, update any `imagePullSecrets` or other references that still point to the old SA name. |
| 4 | **Secret type is restricted** – trying to read a `kubernetes.io/service-account-token` secret that the API server hides by default. | Since Kubernetes 1.24 the `serviceaccount` token secret is not automatically created, and even when it exists the API may mask its data for non‑privileged users. | ```bash\nkubectl get secret <secret-name> -n kubexplain-eval -o yaml\n```<br>Check the `type:` field. If it is `kubernetes.io/service-account-token`, the data fields (`token`, `ca.crt`) are only visible to users with `secrets` `get` permission **and** the `system:serviceaccounts` group. | If the application really needs the token, give it the `system:serviceaccount` group via a `ClusterRole` that includes `secrets` `get`. Usually the better pattern is to let the pod use its own mounted token instead of reading another SA’s token. |
| 5 | **NetworkPolicy or Admission webhook blocking the API request** – rare, but a policy could deny egress to the API server or an admission webhook could reject secret‑list calls. | A `NetworkPolicy` that denies egress from the pod’s namespace to the API server IP/port will cause the request to fail with a timeout, which can be mis‑interpreted as a permission error. | ```bash\nkubectl get networkpolicy -n kubexplain-eval -o yaml\n```<br>Look for egress rules that restrict traffic to `kubernetes.default.svc`. <br>Also check the API‑server logs for “admission webhook denied” messages (requires cluster‑admin access). | Adjust the `NetworkPolicy` to allow egress to the API server (`port: 443`, `to: {podSelector: {}, namespaceSelector: {}}`). If an admission webhook is the culprit, modify its configuration or add an exemption for this SA. |

### How to prioritize the investigation

1. **RBAC missing** – this is the most common cause for “cannot list secrets” errors. The `kubectl auth can-i` test will tell you instantly.
2. **Wrong namespace** – easy to overlook if the code builds the secret name without a namespace.
3. **Token not present/expired** – shows up as authentication failures rather than pure RBAC denials.
4. **Secret type / data masking** – only relevant if the secret you are trying to read is a service‑account token.
5. **NetworkPolicy / webhook** – uncommon, but worth checking if the first three checks all pass.

When you regain access to the cluster, run the verification commands in the order above; the first one that returns a negative result points directly to the root cause, and applying the corresponding fix will restore secret‑listing capability for `s06‑limited‑sa`.
