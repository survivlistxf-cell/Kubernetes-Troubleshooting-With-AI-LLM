## Most likely reasons the `s06‑limited‑sa` service account cannot list Secrets  

| # | Likely cause (probability high → low) | How to verify (when you have cluster access) | How to fix |
|---|---------------------------------------|----------------------------------------------|------------|
| 1 | **Missing RBAC permission** – no `Role`/`ClusterRole` that grants `list` (or `get`) on `secrets` in the `kubexplain-eval` namespace. | ```bash\nkubectl auth can-i list secrets --as=system:serviceaccount:kubexplain-eval:s06-limited-sa -n kubexplain-eval\n```<br>If the command returns `no`, the service account is not allowed to list secrets. <br>Also inspect existing Roles: <br>```bash\nkubectl get role -n kubexplain-eval -o yaml | grep -i secrets\n``` | Create (or update) a Role that includes the needed verbs and bind it to the SA: <br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: Role\nmetadata:\n  name: secret‑reader\n  namespace: kubexplain-eval\nrules:\n- apiGroups: [\"\"]\n  resources: [\"secrets\"]\n  verbs: [\"get\",\"list\"]\n---\napiVersion: rbac.authorization.k8s.io/v1\nkind: RoleBinding\nmetadata:\n  name: secret‑reader‑binding\n  namespace: kubexplain-eval\nsubjects:\n- kind: ServiceAccount\n  name: s06-limited-sa\n  namespace: kubexplain-eval\nroleRef:\n  kind: Role\n  name: secret‑reader\n  apiGroup: rbac.authorization.k8s.io\n```<br>Apply with `kubectl apply -f <file>.yaml`. |
| 2 | **RoleBinding points at the wrong ServiceAccount or wrong namespace** – a Role exists but the binding references a different SA name or a different namespace, so the intended SA never gets the permissions. | ```bash\nkubectl get rolebinding -n kubexplain-eval -o yaml | grep -A5 \"s06-limited-sa\"\n```<br>Check that `subjects[].name` is exactly `s06-limited-sa` and `subjects[].namespace` is `kubexplain-eval`. | Edit the existing RoleBinding (or create a new one) so that the `subjects` field correctly references `system:serviceaccount:kubexplain-eval:s06-limited-sa`. Example edit: <br>```bash\nkubectl edit rolebinding <binding-name> -n kubexplain-eval\n``` |
| 3 | **Attempting to list Secrets in a different namespace** – the code may be calling the API without specifying the namespace, defaulting to `default` or another namespace where the SA has no rights. | Look at the application code or the failing API request (e.g., logs). If the request URL is `/api/v1/secrets` without a namespace segment, it is a cluster‑wide list. Verify with: <br>```bash\nkubectl auth can-i list secrets --as=system:serviceaccount:kubexplain-eval:s06-limited-sa\n``` (no `-n` flag). | Either: <br>• Scope the request to the correct namespace (`/api/v1/namespaces/kubexplain-eval/secrets`). <br>• Or grant the SA a **ClusterRole** that allows `list` on secrets across all namespaces (only if that broader access is acceptable). Example ClusterRole + ClusterRoleBinding: <br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: ClusterRole\nmetadata:\n  name: secret‑cluster‑reader\nrules:\n- apiGroups: [\"\"]\n  resources: [\"secrets\"]\n  verbs: [\"get\",\"list\"]\n---\napiVersion: rbac.authorization.k8s.io/v1\nkind: ClusterRoleBinding\nmetadata:\n  name: secret‑cluster‑reader‑binding\nsubjects:\n- kind: ServiceAccount\n  name: s06-limited-sa\n  namespace: kubexplain-eval\nroleRef:\n  kind: ClusterRole\n  name: secret‑cluster‑reader\n  apiGroup: rbac.authorization.k8s.io\n``` |
| 4 | **Admission‑controller or PSP restrictions that block secret access** – some clusters run a custom admission webhook that denies secret reads for low‑privilege accounts. | Check the cluster’s admission plugins (requires cluster‑admin). <br>If you have access to the API server config or can ask a cluster admin, look for plugins like `NodeRestriction`, `PodSecurityPolicy`, or custom webhooks that filter `secrets`. | If such a plugin is the cause, you need to adjust its configuration or add an exemption for the SA. This typically requires cluster‑admin privileges; coordinate with the platform team. |
| 5 | **Secret is of type `kubernetes.io/service-account-token` and the API server hides its data** – the SA can list the secret object but the `data` field is omitted for non‑privileged users, which may be interpreted as “cannot list”. | Run the list command as the SA and inspect the output: <br>```bash\nkubectl get secret -n kubexplain-eval -o yaml --as=system:serviceaccount:kubexplain-eval:s06-limited-sa\n```<br>If you see the secret objects but their `data` fields are empty, this is the behavior. | No RBAC change needed; the SA already has permission to see the secret metadata. If the application needs the token data, grant it the `secrets` `get` permission (already covered by #1) and ensure the secret is not a “service‑account token” that is automatically projected (use a regular `Opaque` secret instead). |

### Quick verification checklist (run when you regain access)

1. **RBAC check** – `kubectl auth can-i list secrets …` (should return `yes`).  
2. **Role/RoleBinding inspection** – confirm a Role with `resources: ["secrets"]` and `verbs: ["list","get"]` exists and is bound to `s06-limited-sa`.  
3. **Namespace correctness** – verify the code is requesting secrets in `kubexplain-eval` (or grant cluster‑wide access if intentional).  
4. **Admission plugins** – ask a cluster admin if any custom webhook or PSP is limiting secret reads.  
5. **Secret type** – list the secret as the SA and see whether the `data` field is present.

Fix the first issue you encounter; in most clusters the root cause is simply **missing RBAC permission** (cause #1). Once the appropriate Role/RoleBinding is applied, the service account should be able to list the needed secrets. If the problem persists after that, move down the list.
