## Most Likely root‑cause hierarchy for pod **s07‑missing‑cm** (namespace **kubexplain‑eval**)

| # | Probable cause | Why it’s likely (based on the pod name & typical failure patterns) |
|---|----------------|-------------------------------------------------------------------|
| 1 | **Missing or unreadable ConfigMap** referenced by the pod (e.g. as a volume mount or envFrom) | The pod name itself contains “missing‑cm”, which is a strong hint that the manifest expects a ConfigMap that is not present or has a typo. When a required ConfigMap cannot be projected, the container will stay in **Waiting** with a reason such as `CreateContainerConfigError` or `CrashLoopBackOff`. |
| 2 | **Image‑pull failure** (wrong image name, missing tag, private registry auth) | Even if the ConfigMap exists, a bad image reference is the second‑most common reason a container never starts. The pod will show `ErrImagePull` / `ImagePullBackOff`. |
| 3 | **Insufficient resources** (CPU / memory requests > node capacity) | If the scheduler placed the pod but the kubelet cannot allocate the requested resources, the container will stay in `Pending` → `FailedScheduling` or later in `Waiting` with `InsufficientMemory` / `InsufficientCPU`. |
| 4 | **Invalid pod spec** (e.g. wrong field, immutable field change, missing required field) | A malformed spec can cause the API server to reject the pod creation, or the kubelet to report `CreateContainerConfigError`. |
| 5 | **Node‑level problems** (node NotReady, taints, missing runtime) | The pod may be scheduled onto a node that cannot run containers (e.g., kubelet stopped, disk pressure). The pod will stay in `Pending` or `Running` → `CrashLoopBackOff` with events like `NodeNotReady`. |
| 6 | **Security / admission webhook rejection** (PodSecurityPolicy, OPA/Gatekeeper) | If a webhook blocks the pod after creation, you may see a `FailedCreate` event and the pod will never transition to `Running`. |
| 7 | **Finalizer / termination issue** (unlikely for start‑up, but can block pod from being recreated) | If a previous pod with the same name is stuck in `Terminating`, a new pod may never be created. |

## How to verify each cause (once you have cluster access)

### 1. Missing ConfigMap
```bash
# Show the pod’s spec and look for ConfigMap references
kubectl get pod s07-missing-cm -n kubexplain-eval -o yaml | grep -i configmap -A2

# List ConfigMaps in the namespace
kubectl get cm -n kubexplain-eval
```
**What to look for**  
- `CreateContainerConfigError` or `ConfigMap "…" not found` in `kubectl describe pod`.  
- Volume mount path missing or envFrom error.

**Fix**  
- Create the missing ConfigMap (or correct the name) and re‑apply:  
  ```bash
  kubectl create configmap <name> --from-file=... -n kubexplain-eval
  ```
- If the ConfigMap is supposed to be generated by another controller, ensure that controller is running and has the proper RBAC.

### 2. Image‑pull failure
```bash
kubectl describe pod s07-missing-cm -n kubexplain-eval | grep -i "Image"
```
**What to look for**  
- Events: `Failed to pull image`, `ErrImagePull`, `ImagePullBackOff`.  
- Check the image name/tag and registry credentials (`kubectl get secret` for imagePullSecrets).

**Fix**  
- Correct the image reference in the Deployment/Pod manifest.  
- Push the missing tag to the registry or add a valid `imagePullSecret`.  
- Test pulling manually on a node (`docker pull <image>` or `crictl pull <image>`).

### 3. Insufficient resources
```bash
kubectl describe pod s07-missing-cm -n kubexplain-eval | grep -i "Insufficient"
kubectl get nodes -o wide
```
**What to look for**  
- Events: `FailedScheduling` with reason `Insufficient cpu` or `memory`.  
- Pod’s `requests`/`limits` exceeding node capacity.

**Fix**  
- Lower the resource requests/limits, or add more nodes / increase node capacity.  
- Use `kubectl top node` (if metrics‑server is installed) to see current usage.

### 4. Invalid pod spec
```bash
kubectl get pod s07-missing-cm -n kubexplain-eval -o yaml
```
**What to look for**  
- Syntax errors reported by the API server (`kubectl describe pod` will show `FailedCreate`).  
- Immutable field changes (e.g., trying to change `spec.containers[0].name` on an existing pod).

**Fix**  
- Delete the broken pod and recreate it from a corrected manifest (or edit the Deployment).  
- Validate the YAML with `kubectl apply --dry-run=client -f <file>` before re‑applying.

### 5. Node‑level problems
```bash
kubectl get pod s07-missing-cm -n kubexplain-eval -o wide   # shows the node name
kubectl describe node <node-name>
```
**What to look for**  
- Node status `NotReady`, `DiskPressure`, `MemoryPressure`.  
- Events on the node indicating kubelet issues.

**Fix**  
- Resolve node health (restart kubelet, free disk space, fix network).  
- If the node is tainted, add a toleration to the pod or move the pod to a healthy node (e.g., by cordoning the bad node).

### 6. Security / admission webhook rejection
```bash
kubectl describe pod s07-missing-cm -n kubexplain-eval | grep -i "admission"
```
**What to look for**  
- Events with `FailedCreate` and a message from a webhook (e.g., OPA, Gatekeeper).  

**Fix**  
- Adjust the pod spec to satisfy the policy (add required labels, drop prohibited capabilities).  
- If the webhook is mis‑configured, update or temporarily disable it (requires cluster admin rights).

### 7. Finalizer / termination block
```bash
kubectl get pod s07-missing-cm -n kubexplain-eval -o yaml | grep -i finalizers
```
**What to look for**  
- Stuck finalizer entries preventing deletion.  

**Fix**  
- Patch the pod to remove the finalizer (if safe):  
  ```bash
  kubectl patch pod s07-missing-cm -n kubexplain-eval -p '{"metadata":{"finalizers":null}}' --type=merge
  ```
- Investigate the controller that adds the finalizer and fix its logic.
