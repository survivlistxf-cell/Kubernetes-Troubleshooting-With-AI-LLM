**Q:** My HorizontalPodAutoscaler 's12-hpa' in namespace 'kubexplain-eval' does not scale anything. What is the root cause and how do I fix it? I cannot access the cluster right now, so list the most likely causes in order of probability and explain how to verify and fix each one.

**Expected:** HPA targets a Deployment that does not exist (FailedGetScale); fix scaleTargetRef

---

## Most likely reasons why **s12‑hpa** is not scaling (ordered by how often they bite)

| # | Reason (why scaling fails) | How to verify it (when you regain access) | How to fix it |
|---|----------------------------|-------------------------------------------|--------------|
| 1 | **Metrics are unavailable** – the HPA cannot read the metric it is supposed to watch (CPU, memory, custom metric). This is the single biggest cause of a “quiet” HPA. | ```bash\nkubectl get hpa s12-hpa -n kubexplain-eval -o yaml\n```Look at the `status.conditions` section. If you see `type: AbleToScale` or `type: ScalingActive` with `status: "False"` and a message like *“metrics not available”* or *“failed to get cpu utilization”*, the metrics pipeline is broken. Also run `kubectl top pods -n kubexplain-eval` – if it returns an error or empty output, the **metrics‑server** (or custom‑metrics‑adapter) is not working. | • Ensure the **metrics‑server** (or your custom‑metrics adapter) is deployed and healthy (`kubectl get pods -n kube-system -l k8s-app=metrics-server`). <br>• Check that the APIService `v1beta1.metrics.k8s.io` is `Available` (`kubectl get apiservice`). <br>• If you use custom metrics, verify the adapter’s Service and its `ServiceMonitor`/`Prometheus` scrape. <br>• Re‑install or upgrade the metrics‑server version that matches your Kubernetes version. |
| 2 | **Target utilization is already satisfied** – the current CPU/memory usage is below the threshold, so the HPA has no reason to scale. | ```bash\nkubectl top pods -n kubexplain-eval\nkubectl describe hpa s12-hpa -n kubexplain-eval\n```The `describe` output shows the *current* metric value and the *desired* replica count. If `currentCPUUtilizationPercentage` (or the custom metric) is well under the target, scaling will not happen. | • Adjust the target value (e.g., raise `targetCPUUtilizationPercentage` from 50 % to 70 %). <br>• If the workload is genuinely under‑utilized, consider lowering the `minReplicas` instead of forcing scaling. |
| 3 | **MinReplicas = MaxReplicas** – the HPA has no headroom to increase or decrease the replica count. | ```bash\nkubectl get hpa s12-hpa -n kubexplain-eval -o jsonpath='{.spec.minReplicas} {.spec.maxReplicas}'\n```If both numbers are the same, the HPA cannot change the replica count. | • Edit the HPA to give a wider range, e.g. `kubectl edit hpa s12-hpa -n kubexplain-eval` and set `spec.minReplicas: 2` and `spec.maxReplicas: 10`. |
| 4 | **Scale target is not a scalable workload** – the HPA points at a resource that does not support scaling (e.g., a `Pod` or a `Job`). | ```bash\nkubectl get hpa s12-hpa -n kubexplain-eval -o yaml | grep targetRef -A3\n```The `targetRef.kind` must be one of `Deployment`, `ReplicaSet`, `StatefulSet`, or `ReplicationController`. If it is `Pod` or something else, the controller will ignore the HPA. | • Change the `targetRef.kind` to a supported controller and ensure the `name` matches an existing object. <br>• Re‑apply the corrected HPA manifest. |
| 5 | **RBAC prevents the HPA controller from reading metrics or scaling the target** – the `system:controller:hpa` service account lacks the required `get`, `list`, `watch`, `update` permissions. | ```bash\nkubectl auth can-i get hpa -n kubexplain-eval --as system:serviceaccount:kube-system:horizontal-pod-autoscaler\nkubectl auth can-i update deployment/<target-name> -n kubexplain-eval --as system:serviceaccount:kube-system:horizontal-pod-autoscaler\n```If any of these return “no”, the controller is blocked. | • Add or fix the ClusterRoleBinding for the HPA controller (usually `system:controller:hpa`). <br>• Example: <br>```yaml\napiVersion: rbac.authorization.k8s.io/v1\nkind: ClusterRoleBinding\nmetadata:\n  name: hpa-controller\nsubjects:\n- kind: ServiceAccount\n  name: horizontal-pod-autoscaler\n  namespace: kube-system\nroleRef:\n  kind: ClusterRole\n  name: system:controller:hpa\n  apiGroup: rbac.authorization.k8s.io\n```Apply it and restart the HPA controller pod if needed. |
| 6 | **Cluster‑wide resource quota or limit range blocks scaling** – a `ResourceQuota` caps the number of pods in the namespace, so the HPA cannot create more replicas. | ```bash\nkubectl get quota -n kubexplain-eval\nkubectl describe quota <quota-name> -n kubexplain-eval\n```If `used` pods = `hard` pods, scaling will be denied. | • Increase the quota (`spec.hard.pods`) or delete/adjust the quota. <br>• Alternatively, lower the HPA `maxReplicas` to stay within the quota. |
| 7 | **HPA controller itself is not running or is unhealthy** – the `horizontal-pod-autoscaler` deployment (or the controller manager) has been scaled down or crashed. | ```bash\nkubectl get deployment -n kube-system horizontal-pod-autoscaler\nkubectl get pods -n kube-system -l k8s-app=horizontal-pod-autoscaler\n```If there are no ready pods, the controller cannot act. | • Scale the deployment back up (`kubectl scale deployment horizontal-pod-autoscaler -n kube-system --replicas=1`). <br>• Check its logs for errors (`kubectl logs <pod> -n kube-system`). |
| 8 | **Stabilization window / cooldown prevents immediate scaling** – the HPA’s `behavior.scaleUp.stabilizationWindowSeconds` (or the default 5 min) is holding the replica count steady after a recent change. | ```bash\nkubectl describe hpa s12-hpa -n kubexplain-eval | grep -A2 stabilization\n```If you see a non‑zero window, recent scaling events may be ignored until it expires. | • Reduce the stabilization window or `scaleUp`/`scaleDown` policies in the HPA spec. <br>• Or wait for the window to elapse before testing again. |
| 9 | **Pod template cannot be instantiated** – the underlying Deployment fails to create new Pods (image pull error, insufficient node resources, node selector mismatch). The HPA thinks it can scale, but the Deployment stays at the old replica count. | ```bash\nkubectl get deployment <target-name> -n kubexplain-eval -o yaml | grep replicas\nkubectl describe deployment <target-name> -n kubexplain-eval | grep -A5 Events\n```Look for `FailedCreate` or `Insufficient cpu` events. | • Fix the root cause (correct image name/tag, add needed node selectors, increase node capacity, adjust resource requests/limits). <br>• Once the Deployment can schedule Pods, the HPA will be able to increase replicas. |
|10| **Incorrect HPA version / API mismatch** – using an older HPA spec (`autoscaling/v1`) that does not support custom metrics while the manifest references them, causing the controller to ignore the metric. | ```bash\nkubectl get hpa s12-hpa -n kubexplain-eval -o yaml | grep apiVersion\n```If it’s `autoscaling/v1` but you have `metrics` fields, they are silently dropped. | • Upgrade the HPA manifest to `autoscaling/v2beta2` (or `v2`) which supports custom metrics. <br>• Re‑apply the corrected manifest. |

### Quick verification checklist (run when you can log in)

1. **Inspect the HPA status** – look for `conditions` and the *desired* replica count.  
2. **Confirm metrics availability** – `kubectl top pods` and the health of `metrics-server` (or custom‑metrics adapter).  
3. **Validate the scaling range** – `minReplicas` vs `maxReplicas`.  
4. **Check the target reference** – ensure it points to a scalable controller (`Deployment`, `StatefulSet`, …).  
5. **Review RBAC** – the HPA controller’s service account must be able to `get`/`list` metrics and `update` the target.  
6. **Look for quota limits** – `kubectl get quota` in the namespace.  
7. **Make sure the HPA controller is alive** – pods in `kube-system` with label `k8s-app=horizontal-pod-autoscaler`.  
8. **Examine recent scaling events** – see if a stabilization window is holding the count steady.  
9. **Inspect the target workload’s events** – any pod‑creation failures will stop scaling even if the HPA requests more replicas.  
10. **Confirm API version** – use `autoscaling/v2*` if you need custom metrics.

Address the first failing check you encounter; in most clusters the root cause is **missing or broken metrics collection** (Cause #1). Fixing that usually restores normal autoscaling behavior. Good luck!
