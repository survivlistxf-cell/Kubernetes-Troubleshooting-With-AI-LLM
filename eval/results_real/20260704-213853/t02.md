## Prompt 1 (fără dovezi) — 75.7s
**User:** I updated the frontend of my e-learning application and now the new frontend pod in namespace 'elearning' never becomes Ready. What is wrong?

**AI:** ### What to look at first

1. **Pod phase and conditions** – `kubectl get pod -n elearning` will show the high‑level phase (Pending, Running, etc.) and the **Ready** column.  
2. **Detailed status** – `kubectl describe pod <pod‑name> -n elearning` lists:
   * **Pod conditions** (`Ready`, `ContainersReady`, `Initialized`, …) and the timestamps when they changed.  
   * **Events** that often contain the root cause (image‑pull failures, scheduling problems, probe failures, etc.).  
3. **Container state** – In the `describe` output you’ll see each container’s state (`Waiting`, `Running`, `Terminated`) and the **Reason** field (e.g. `ImagePullBackOff`, `CrashLoopBackOff`, `CreateContainerConfigError`).  
4. **Readiness/Liveness probes** – If a readiness probe is defined and keeps failing, the pod will stay in `Ready=False` even though the containers are running. Check the probe configuration in the pod spec and the probe logs (`kubectl logs <pod‑name> -c <container> -n elearning`).  
5. **Resource constraints** – Insufficient CPU/memory can cause the kubelet to keep the container in `Waiting` (reason `InsufficientMemory`/`InsufficientCPU`).  
6. **Image issues** – Wrong image name, tag, or missing pull secret leads to `ImagePullBackOff` or `ErrImagePull`.  

### Typical reasons a pod never becomes Ready

| Symptom (from `kubectl get pod`) | Likely cause (from `describe`/`logs`) |
|-----------------------------------|----------------------------------------|
| `Pending` with no node assigned   | No node matches the pod’s **nodeSelector**, **affinity**, or **taints/tolerations**. |
| `Pending` with `ContainerCreating` | Image pull problem, missing secret, or network issue reaching the registry. |
| `Running` but `Ready` = `False`  | Failing **readiness probe**, or container is still initializing (e.g., startup probe not succeeded). |
| `CrashLoopBackOff`                | Container exits repeatedly (application error, missing config, wrong command). |
| `Error` or `CreateContainerConfigError` | Invalid pod spec (bad env var, volume mount, securityContext, etc.). |
| `Terminating` stuck               | Finalizers preventing deletion, or a sidecar container that never exits. |

### How to narrow it down

1. **Get the pod name**  
   ```bash
   kubectl get pods -n elearning
   ```

2. **Inspect the pod**  
   ```bash
   kubectl describe pod <pod-name> -n elearning
   ```
   * Look for the **Conditions** section – which condition is `False` and why?  
   * Review the **Events** at the bottom – they often contain the exact error message.

3. **Check container logs** (especially the main container that serves the frontend)  
   ```bash
   kubectl logs <pod-name> -c <container-name> -n elearning
   ```
   * If the container never starts, you may see nothing or an early error.  
   * If the container crashes, the logs will show the stack trace or exit code.

4. **Verify probes** (if any)  
   * In the pod spec, note `readinessProbe` / `livenessProbe` / `startupProbe`.  
   * If they use HTTP, exec, or TCP checks, try the same command locally inside the container (`kubectl exec -it <pod-name> -c <container> -- <probe‑command>`).  
   * Adjust `initialDelaySeconds`, `periodSeconds`, or `failureThreshold` if the application needs more time to start.

5. **Confirm resource limits**  
   * If the pod requests more CPU/memory than any node can provide, it will stay pending.  
   * Check node allocatable resources (`kubectl describe node <node>`).

6. **Validate image and pull secret**  
   * Ensure the image name/tag exists in the registry.  
   * If the registry is private, verify the secret is referenced correctly (`imagePullSecrets`).

| **Network / DNS issue** | The node (`worker-1`) cannot reach the registry (e.g., firewall, proxy, DNS). |
