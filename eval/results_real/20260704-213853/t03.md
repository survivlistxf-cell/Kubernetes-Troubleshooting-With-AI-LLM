**AI:** ### Why a Pod can stay **Pending**

When a Pod is in the **Pending** phase the scheduler has not been able to bind it to any node. The most common reasons are:

- **Insufficient resources** ‚Äì the cluster does not have enough free CPU or memory (or other requested resources such as GPUs, hugepages, etc.) to satisfy the pod‚Äôs **resource requests**.  
- **Node‚Äëlevel constraints** ‚Äì the pod may have a **node selector**, **node affinity**, **taints/tolerations**, or **pod affinity/anti‚Äëaffinity** that no node currently matches.  
- **HostPort / hostNetwork conflicts** ‚Äì if the pod uses a `hostPort` (or `hostNetwork`) the scheduler can only place the pod on a node where that port is free.  
- **PersistentVolume binding** ‚Äì for stateful workloads (e.g., a PostgreSQL pod) the pod may be waiting for its PVC to bind to a PV. If the requested storage class cannot provision a volume, the pod stays pending.  
- **Missing or invalid image** ‚Äì the scheduler can bind the pod, but the kubelet will later report a ‚Äúpull back‚Äëoff‚Äù. In that case the pod usually moves to **ContainerCreating**, not Pending, but a malformed `imagePullPolicy` combined with a `nodeSelector` that never matches can keep it stuck.  
- **Admission‚Äëcontroller rejections** ‚Äì a validating webhook or other admission controller may reject the pod creation, leaving it unscheduled.  
- **Pod disruption / eviction** ‚Äì if the pod is part of a StatefulSet and a previous pod is terminating but still holds a PVC, the new pod may wait for the previous one to finish.

### What to look at for `postgres-0` in the `elearning` namespace

1. **Scheduler messages** ‚Äì `kubectl describe pod postgres-0 -n elearning` will show events such as:
   - ‚Äú0/3 nodes are available: 3 Insufficient cpu, 3 Insufficient memory‚Ä¶‚Äù
   - ‚Äú0/3 nodes are available: 1 node(s) didn‚Äôt match node selector, 2 node(s) had taint ‚Ä¶‚Äù
   - ‚Äúpersistentvolumeclaim ‚Äúpostgres-data‚Äù not bound‚Äù

2. **Resource requests/limits** ‚Äì PostgreSQL pods often request a few hundred millicores and a gigabyte of RAM. If you recently increased those values, the scheduler may now find no node with enough free capacity.

3. **PVC status** ‚Äì Check the PVC used by the pod (`kubectl get pvc -n elearning`). If it is still `Pending`, the pod cannot start because the volume is not ready.

4. **Node constraints** ‚Äì Verify any `nodeSelector`, `affinity`, or `tolerations` in the pod spec. A recent change (e.g., adding `nodeSelector: disktype: ssd`) could filter out all nodes.

5. **Taints** ‚Äì Run `kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name} {.spec.taints}\n{end}'` to see if nodes are tainted with something the pod does not tolerate (e.g., `node.kubernetes.io/not-ready`).

6. **HostPort usage** ‚Äì If the pod declares a `hostPort` (common for PostgreSQL when exposing the DB on the host), only one pod can use that port per node. If every node already has a pod using that port, the scheduler cannot place a new one.

### Typical next steps

- **Reduce or adjust resource requests** (or add more nodes / larger nodes).  
- **Fix PVC binding** ‚Äì ensure the storage class can provision a volume, or manually bind a suitable PV.  
- **Remove or modify node selectors / affinity** that are too restrictive.  
- **Add tolerations** for any taints that are preventing scheduling.  
- **Check for hostPort conflicts** and, if possible, switch to a Service + ClusterIP instead of binding the port on the host.

Once the underlying scheduling blocker is resolved, the pod will transition from **Pending** to **Running**, and your e‚Äëlearning application will be able to reach the PostgreSQL database again.

### How to fix it

#### 1. Remove or correct the node selector
If the selector was added by mistake, edit the StatefulSet (or the pod template) to drop it:

```bash
kubectl edit statefulset postgres -n elearning
```

Locate the `nodeSelector:` block and either delete it or change it to a label that actually exists on a node, e.g.:

```yaml

  kubernetes.io/hostname: worker-1   # or worker-2
```

#### 2. Add appropriate tolerations (if you really want the pod to run on a tainted node)
