* Kubernetes restarts the pod, sees the non‑zero exit, and puts the pod into **CrashLoopBackOff**.

## How to verify the exact problem

### 1. Inspect the pod’s environment (where the DB connection info lives)

```bash
kubectl describe pod backend-7bfb848455-8s59g -n elearning
```

* Look for `env:` entries such as `SPRING_DATASOURCE_URL`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD` (often sourced from a `Secret` or `ConfigMap`).  
* Confirm that the URL points to the correct service name (`postgres`), port (`5432`), and database name.

### 2. Check the Secret / ConfigMap that provides the credentials

```bash
# If a secret is used
kubectl get secret <secret-name> -n elearning -o yaml

# If a ConfigMap is used
kubectl get configmap <configmap-name> -n elearning -o yaml
```

* Verify that the keys match what the application expects (`username`, `password`, etc.).  
* Ensure the secret was not accidentally overwritten or deleted during the config change.

### 3. Test connectivity from inside the backend pod

```bash
kubectl exec -it backend-7bfb848455-8s59g -n elearning -- /bin/sh
# inside the container
apk add --no-cache postgresql-client   # (or use busybox nc if psql not available)
pg_isready -h postgres -p 5432 -U <user> -d <db>
# or a simple TCP test
nc -zv postgres 5432
```

* A successful connection proves the network path and service name are correct.  
* Failure points to DNS, Service, or network‑policy issues.

### 4. Verify the PostgreSQL service and pod are healthy

```bash
kubectl get svc postgres -n elearning
kubectl get pod postgres-0 -n elearning
kubectl logs postgres-0 -n elearning
```

* The service should have a ClusterIP and port `5432`.  
* The PostgreSQL pod must be `Running` with no recent restarts.

### 5. Look for recent changes that could have broken the DB config

* Review the Git commit or `kubectl apply` that was performed just before the crash.  
* Typical culprits:
  * Changed `SPRING_DATASOURCE_URL` from `jdbc:postgresql://postgres:5432/elearning` to an incorrect host or missing `jdbc:` prefix.  
  * Updated the secret name but forgot to update the Deployment’s `envFrom` reference.  
  * Added a NetworkPolicy that blocks traffic from the `backend` pods to the `postgres` pod.

## Common fixes

| Symptom | Fix |
|---------|-----|
| **Wrong JDBC URL** (e.g., `postgres:5432` instead of `jdbc:postgresql://postgres:5432/elearning`) | Edit the Deployment (or ConfigMap) to use the correct `jdbc:postgresql://<service>:<port>/<db>` format. |
| **Missing/incorrect credentials** | Re‑create or update the Secret with the proper `username`/`password` keys, and ensure the Deployment references the correct secret name. |
| **Service name typo** | Verify the Service object name (`postgres`) matches the host part of the URL. |
| **NetworkPolicy blocking traffic** | Add an ingress rule allowing traffic from the `backend` namespace/pod selector to port `5432` on the `postgres` pod. |
| **PostgreSQL pod not ready** | Wait for the DB pod to become Ready, or investigate its own logs for startup failures. |

After correcting the configuration, roll out the change:

```bash
kubectl rollout restart deployment/backend -n elearning
```

Then watch the pods:

```bash
kubectl get pods -n elearning -w
```

You should see the backend pods transition to **Running** without entering `CrashLoopBackOff`.

## TL;DR

Fix the datasource settings (or any NetworkPolicy that blocks it), then restart the deployment. Once the DB connection succeeds, the bean creation will complete and the pod will stay up.
