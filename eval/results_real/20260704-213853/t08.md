**User:** The backend pod of my e-learning application in namespace 'elearning' keeps getting killed and restarted every few minutes. Why?

**AI:** ### What “killed and restarted every few minutes” usually means

When a Pod repeatedly stops and starts, the kubelet is applying the **restart policy** for the containers inside the pod. The most common reasons are:

| Symptom | Typical cause | What the kubelet does |
|---------|---------------|-----------------------|
| **CrashLoopBackOff** (you see the pod status `CrashLoopBackOff` after a few restarts) | The container exits with a non‑zero code, or is killed by the runtime (e.g., OOMKilled). | The kubelet restarts the container, applying an exponential back‑off delay that grows after each failure. |
| **OOMKilled** (event shows `OOMKilled`) | The container exceeds its memory limit. | The runtime kills the container; the pod is then restarted according to its `restartPolicy`. |
| **Liveness probe failure** | A liveness probe repeatedly returns failure. | The kubelet kills the container and restarts it, regardless of the exit code. |
| **Startup probe failure** (if defined) | The container never passes the startup probe within the timeout. | The container is killed and restarted according to the pod’s `restartPolicy`. |
| **Readiness probe failure** (does **not** cause a restart) | The pod stays “NotReady” but will not be killed. | Only affects service routing; not a restart cause. |
| **Sidecar‑triggered in‑place restart** (RestartAllContainers feature) | A sidecar container exits with a special exit code that matches a `restartPolicyRules` entry (e.g., exit code 88). | All containers in the pod are terminated and started again, preserving the pod’s UID, IP, and volumes. |
| **Node pressure / eviction** | Node runs out of memory or disk, or the pod is evicted by the scheduler. | The pod is terminated and later rescheduled on another node (new UID). |
| **Manual deletion / controller recreation** | A higher‑level controller (Deployment, ReplicaSet, Job, etc.) deletes the pod and creates a new one. | The pod gets a fresh UID; you’ll see a new pod name each time. |

### How to pinpoint the exact cause for the `elearning` pod

1. **Check the pod’s current status and recent events**  
   Look at the `STATUS` column (`kubectl get pod -n elearning`) and then run `kubectl describe pod <pod-name> -n elearning`.  
   - The **Events** section will show messages such as `Killing`, `BackOff`, `OOMKilled`, `Failed`, `FailedScheduling`, or a custom event from a sidecar.  
   - The **Containers** subsection will list each container’s **State**, **Last State**, **Reason**, and **Exit Code**.

2. **Inspect container exit codes and reasons**  
   - `Exit Code 0` → normal termination (should not be restarted unless `restartPolicy: Always`).  
   - Non‑zero exit codes → failure; the pod’s `restartPolicy` determines whether it restarts.  
   - `Reason: OOMKilled` → memory limit exceeded.  
   - `Reason: Error` with a custom message → application‑level crash.

3. **Look at probe configurations**  
   - If a **liveness** or **startup** probe is defined, the `describe` output will show the last probe result and any failures.  
   - Repeated failures will appear as `Liveness probe failed` or `Startup probe failed`, which directly cause restarts.

4. **Check resource usage**  
   - `kubectl top pod <pod-name> -n elearning` (if metrics‑server is installed) can reveal whether the pod is hitting its CPU or memory limits.  
   - If you see memory usage spiking right before a restart, OOMKilled is likely.

5. **Examine sidecar behavior (if you have one)**  
   - If the pod includes a sidecar with `restartPolicyRules`, the **Events** may contain a line like `RestartAllContainers action triggered by exit code 88`.  
   - In that case the whole pod is restarted in‑place (UID, IP, and volumes stay the same).

6. **Verify the controlling workload**  
   - If the pod is managed by a **Deployment**, **ReplicaSet**, **StatefulSet**, or **Job**, the controller may be recreating the pod after it terminates.  
   - `kubectl get replicaset,deployment,sts,job -n elearning` can show the owner reference.

### Typical next steps once you know the reason

| Identified reason | What to do next |
|-------------------|-----------------|
| **CrashLoopBackOff / non‑zero exit** | Check the container’s logs (`kubectl logs <pod> -n elearning`) to see the application error. Fix the bug, adjust the command, or change the `restartPolicy` if appropriate. |
| **OOMKilled** | Increase the memory `requests`/`limits` in the pod spec, or optimise the application’s memory usage. |
| **Liveness/startup probe failure** | Verify the probe command/URL, adjust `initialDelaySeconds`, `periodSeconds`, `failureThreshold`, or fix the underlying service that the probe checks. |
| **Sidecar‑triggered restart** | Review the sidecar’s exit‑code rules. If the sidecar is exiting intentionally (e.g., a watcher), ensure the exit code matches the intended action, or modify the rule if you don’t want a full pod restart. |
| **Node pressure / eviction** | Add more nodes or increase node resources, or lower the pod’s resource requests/limits. |
| **Controller recreation** | Ensure the controller’s `replicas` count matches the desired state, and that the pod template is correct. If the pod is meant to be a one‑off job, set `restartPolicy: Never` or use a `Job` instead of a `Deployment`. |

### Root cause (specific to your deployment)

**The Spring‑Boot backend container runs out of memory and is terminated (OOMKilled) because its memory limit is set to only 64 MiB, which is far below what a typical Java application needs.**  
When the JVM starts it allocates a heap (by default up to ~¼ of the container’s memory) and then quickly exceeds the 64 MiB cap, so the kernel kills the process (exit 137). The pod then restarts, enters `CrashLoopBackOff`, and repeats indefinitely.

## Fixes – make the container fit its workload

### 1. Increase the memory limits (and requests)

Edit the Deployment (or the pod template) to give the backend a realistic amount of RAM, e.g. 256 MiB – 512 MiB for a small Spring‑Boot app.

```yaml
# backend deployment snippet
resources:
  requests:
    memory: "256Mi"
    cpu: "100m"
  limits:
    memory: "512Mi"
    cpu: "500m"
```

Apply the change:

```bash
kubectl apply -f backend-deployment.yaml
# or, if you edit in‑place:
kubectl set resources deployment/backend \
  -n elearning \
  --limits=memory=512Mi,cpu=500m \
  --requests=memory=256Mi,cpu=100m
```

### 2. Tune the JVM heap to respect the container limit

If you cannot raise the limit (e.g., node‑level constraints), tell the JVM to use a smaller heap:

```bash
# Add to the container env (or command line)
- name: JAVA_TOOL_OPTIONS
  value: "-XX:MaxRAMPercentage=50.0 -XX:InitialRAMPercentage=25.0"
```

*`MaxRAMPercentage`* = percentage of the container’s memory that the JVM may allocate for the heap. With a 64 MiB limit you’d need to set it to ~30 % or lower, but it’s usually easier to raise the limit instead.

### 3. (Optional) Add a **startup probe** so the pod isn’t marked Ready until the app is fully up

```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 30   # wait up to ~5 min
```

This does **not** fix the OOM, but prevents the Service from routing traffic to a pod that is still starting.

### 4. Verify the fix

```bash
# Watch the pods until they become Ready
kubectl get pods -n elearning -w
```

You should see the backend pods reach `Running` / `READY 1/1` and the `RESTARTS` count stop increasing.
