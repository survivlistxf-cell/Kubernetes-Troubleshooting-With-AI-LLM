## Prompt 1 (fără dovezi) — 93.1s
**User:** Requests to my e-learning API at /api/ suddenly return an HTML error page instead of JSON, although all pods in namespace 'elearning' are Running. What is the root cause?

**AI:** **Root cause:** the request is no longer reaching the e‑learning pods because the Service that should expose them has **no endpoints**. When a Service has no matching pod endpoints (often because the pod labels no longer match the Service’s selector), the Ingress controller cannot forward the traffic to the application and falls back to its default backend, which returns a generic HTML error page (e.g. 404/502) instead of the JSON payload your API normally produces.

**Why the pods look healthy**

- `kubectl get pods -n elearning` shows all pods in the *Running* state, so the containers themselves are fine.
- However, the Service that fronts those pods is a separate Kubernetes object. If its selector does not select any pods, the Service’s `ENDPOINTS` list is empty, and the traffic that reaches the Service (or the Ingress that points to it) is rejected by the Ingress controller’s default backend.

**Typical triggers**

- A recent change to pod labels (e.g., `app=elearning` → `app=elearning-api`) without updating the Service selector.
- Modification of the Service selector itself (e.g., typo or version bump).
- Deleting/re‑creating the pods with a different label set while the Service remained unchanged.

**How to verify**

```bash
# Inspect the Service and its selector
kubectl get svc -n elearning <service-name> -o yaml

# List the endpoints the Service currently has
kubectl get endpoints -n elearning <service-name>

# Show the labels on the running pods
kubectl get pods -n elearning -o wide --show-labels
```

If `kubectl get endpoints` shows an empty `subsets` list, and the pod labels do not match the Service’s selector, you have identified the mismatch.

**Fix**

1. Align the Service selector with the pod labels, or
2. Update the pod labels to match the existing Service selector.

After the Service has active endpoints, the Ingress will route traffic to the pods and the API will again return JSON as expected.
