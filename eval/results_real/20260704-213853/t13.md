## Prompt 1 (fără dovezi) — 61.2s
**User:** My e-learning site suddenly returns 404 Not Found from nginx for every page, although all pods in namespace 'elearning' are Running and nothing was redeployed. What is the root cause?

**AI:** ### What a 404 from **nginx** usually means

* **nginx is still running** – the pod is in *Running* state, so the container started successfully.  
* **The request reaches the container** – because the response is an HTTP 404 generated by nginx, the traffic is being routed to the pod (the Service/Ingress is still working).  
* **nginx cannot find the requested file** – a 404 is returned when the file or location defined in the nginx configuration does not exist on the filesystem that nginx is serving.

Given that nothing was redeployed, the most common reasons for this symptom are **runtime changes to the data or configuration that nginx serves**, not a change to the pod spec itself.

#### Typical culprits

| Area | Why it can produce a 404 | What to look for |
|------|--------------------------|------------------|
| **ConfigMap / Secret that provides site files** | If the ConfigMap that is mounted as the web‑root is edited, deleted, or the mount becomes read‑only/empty, the files disappear from the pod’s filesystem. | Check the ConfigMap’s current data (`kubectl get configmap … -o yaml`) and the pod’s volume mounts (`kubectl describe pod …`). Look for events such as “MountVolume.SetUp failed”. |
| **Volume that holds the site content** (e.g., PersistentVolumeClaim) | If the PVC is unbound, the underlying storage is detached, or the mount point is empty, nginx will serve nothing. | Verify the PVC status (`kubectl get pvc`) and the pod’s volume status in `kubectl describe pod`. |
| **nginx configuration (ConfigMap or inline)** | A change to the `nginx.conf` (or a location block) that points to a wrong root directory will cause every request to miss. | Inspect the ConfigMap that supplies `nginx.conf` and the `nginx -T` output (if you can exec into the pod). |
| **File permissions / securityContext** | If the container’s user cannot read the files (e.g., after a securityContext change), nginx will treat the path as missing. | Look at the pod’s `securityContext` and the file ownership inside the container. |
| **Node‑level issue** (disk full, node reboot) | A node that ran out of disk space or was restarted can lose the temporary files that were copied into the container’s image at start‑up. | Check node conditions (`kubectl describe node <node>`) and pod events for “FailedMount” or “OOMKilled”. |

#### How to pinpoint the root cause

1. **Describe one of the pods** in the `elearning` namespace.  
   *Focus on the “Volumes” section and the “Events” list.*  
   Look for any warning or error messages such as `FailedMount`, `MountVolume.SetUp failed`, or `Back-off pulling image` (the latter would indicate a different problem, but it’s useful to confirm the pod really started).

2. **Inspect the ConfigMap(s) that are mounted** (often named something like `elearning‑site‑files` or `nginx‑conf`).  
   Verify that the expected HTML/JS/CSS files are still present in the ConfigMap data. If the ConfigMap was edited to an empty value, the pod will see an empty directory and return 404 for every path.

3. **Check the pod’s filesystem** (if you can exec).  
   Run `ls -l /usr/share/nginx/html` (or whatever `root` is set to). If the directory is empty or the files have wrong permissions, that explains the 404.

4. **Review recent events in the namespace** (`kubectl get events -n elearning`).  
   A recent `Warning` about a ConfigMap update, PVC binding failure, or node condition will give a clue.

#### Most likely root cause for your situation

Because the pods are still **Running** and nothing was redeployed, the **most probable cause is that the ConfigMap (or Secret) that supplies the static site files was modified or deleted**, causing the mount inside the pod to become empty. Nginx therefore has no files to serve and returns a generic 404 for every request.

If the ConfigMap was edited accidentally (e.g., a CI/CD job that updates it with a new version but failed), the pods will instantly see the new (empty or incomplete) content without needing a rollout. Restoring the original ConfigMap data (or re‑applying the correct ConfigMap) will make the files reappear in the pod’s mount and the site will start serving pages again.  

If the ConfigMap looks correct, then move on to the other items in the table (PVC, nginx.conf, permissions, node health) using the same `describe`/`events` checks.

### Root cause in *your* case

**The request is being sent to the Ingress controller without the expected `Host: elearning.example.com` header, so the Ingress rule does not match and the controller serves its default‑backend 404 page.**  

In other words, the external IP you are hitting (`203.25.143.55`) is correctly reaching the NGINX Ingress controller, but because the Ingress object only defines a host‑based rule, the controller does not know which backend to use for a request that contains an IP address (or any other host) in the `Host` header.

## How to verify it yourself

```bash
# 1. Show the Ingress controller Service and its external IP
kubectl -n ingress-nginx get svc ingress-nginx-controller -o wide

# 2. Test the rule with the correct Host header
curl -v -H "Host: elearning.example.com" http://203.25.143.55/

# 3. If you omit the Host header you will see the same 404 you already got
curl -v http://203.25.143.55/
```

You should see the frontend page returned in step 2, confirming that the routing works when the host matches.

## How to fix it

### 1. Use the proper hostname (preferred)

* Create a DNS A‑record (or edit `/etc/hosts` for testing) that points `elearning.example.com` to the public IP (`203.25.143.55` or the external IP of the Ingress Service).  
* Then access the site via `http://elearning.example.com/`.

### 2. Add a catch‑all rule (if you must serve by IP)

If you want the site to be reachable by the bare IP address, add a rule without a `host` field (or with `host: ""`) that points to the same backend, e.g.:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: elearning-ingress
  namespace: elearning
spec:
  ingressClassName: nginx
  rules:
  - host: elearning.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
      - path: /api/
        pathType: Prefix
        backend:
          service:
            name: backend
            port:
              number: 8080
  - http:               # <-- catch‑all rule (no host)
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
```

Apply the updated Ingress and give the controller a minute to reload:

```bash
kubectl apply -f ingress.yaml
```

### 3. (Optional) Set a custom default backend

If you prefer a custom 404 page instead of the generic NGINX one, create a small Service/Deployment that serves a static page and reference it in the Ingress controller’s `--default-backend` flag (or via the `defaultBackend` field of the Ingress object, if your controller supports it).

## Quick checklist to get the site back

1. **DNS / Host header** – ensure `elearning.example.com` resolves to the public IP.  
2. **Test with explicit Host header** – `curl -H "Host: elearning.example.com" http://<public‑IP>/`.  
3. **If you need IP‑only access**, add a host‑less rule (or make the existing rule the default backend).  
4. **Re‑apply the Ingress** and wait a few seconds for the NGINX controller to reload its configuration.  

Once the request’s `Host` header matches a rule, the Ingress will forward traffic to the `frontend` service and the 404 will disappear.
